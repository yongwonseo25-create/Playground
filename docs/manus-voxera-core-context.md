# VOXERA 핵심 배경지식
## Manus 지식 탭 업로드용 기준 문서

## 1. 서비스 한 줄 정의

VOXERA는 음성을 듣고, 핵심을 정리하고, 실제 실행 항목으로 전환하는 **B2B 음성 실행 에이전트**다.

---

## 2. 제품 정체성

VOXERA는 단순 음성 기록 앱이 아니다.  
핵심 목적은 음성을 텍스트로 저장하는 것이 아니라, **실제로 해야 할 일을 빠르게 실행 구조로 바꾸는 것**이다.

핵심 철학:
- Listen: 사용자의 음성 입력을 받는다
- Think: 요약, 액션 아이템, 우선순위, 일정성을 구조화한다
- Act: 실제 실행 보드와 업무 도구에 연결해 행동으로 전환한다

즉, VOXERA는 “기록 도구”보다 **실행 전환 도구**에 가깝다.

---

## 3. 타깃 사용자

주요 타깃:
- 1~20인 규모의 빠른 실행 팀
- CEO / Founder
- Operations Manager
- Sales Manager
- PM
- 실행력이 중요한 한국어 기반 팀

이 사용자들은 긴 문서, 복잡한 세팅, 무거운 CRM/PM 시스템보다  
**빠르게 검토하고 바로 실행하는 구조**를 선호한다.

---

## 4. 제품 목표

VOXERA의 목표는 아래 4가지다.

1. 음성 입력을 빠르게 구조화한다
2. 실제 해야 할 일만 분리한다
3. 실행 보드에 바로 연결한다
4. 문서/일정/알림까지 후속 행동을 이어준다

---

## 5. UX 원칙

### 5-1. 실행 우선
- 모든 화면과 결과물은 “읽기 좋음”보다 “바로 행동 가능함”을 우선한다

### 5-2. 초보자 친화
- 한국 사용자 기준으로 직관적이어야 한다
- 영어 메타 용어, 복잡한 구조, 관리자형 설계는 기본값이 되면 안 된다

### 5-3. 구조는 단순하게
- 받은 것
- 실행할 것
- 끝난 것
이 3단계만 명확하면 된다

### 5-4. 긴 설명보다 짧은 흐름
- 사용자가 “검토 -> 실행 전환 -> 실행” 흐름을 즉시 이해해야 한다

### 5-5. 밝고 정돈된 톤
- UI/대시보드/가이드는 밝은 파스텔톤, 화이트 배경, 부드러운 카드형 구조를 선호한다
- 과한 엔터프라이즈 느낌, 어두운 테마, 복잡한 위키형 문서는 선호하지 않는다

## 6. 현재 제품 아키텍처 핵심

### 6-1. 프론트엔드
- Next.js App Router + TypeScript + Tailwind + shadcn/ui
- 모바일 우선
- 음성 우선
- 상태 머신 중심 구조

### 6-2. 음성 엔진 원칙
- MediaRecorder 사용 금지
- Blob/timeslice 방식 금지
- AudioWorklet + PCM over WSS only

### 6-3. 비용/중복 방어 원칙
- 녹음은 정확히 15초에서 종료
- `clientRequestId`는 async 업로드 전에 동기적으로 생성
- 중복 업로드/중복 과금/중복 제출 방지

### 6-4. 상태 머신 원칙
고정 8상태:
- idle
- permission-requesting
- ready
- recording
- stopping
- uploading
- success
- error

## 7. 현재 툴별 전략

## 7-1. Notion

### 역할
- 사용자가 가장 익숙한 실행 대시보드/검토 공간

### 최종 방향
- Make.com을 Notion 메인 경로에서 제외
- 백엔드가 Notion API로 직접 적재
- 즉: `backend -> Notion direct-write`

### 이유
- 홉 수 감소
- 디버깅 단순화
- 멱등성/중복 방어 통합
- OAuth + template 기반 연결이 공식 경로로 가능

### Notion 구조 원칙
- 메인 화면은 항상 펼쳐진 홈 대시보드
- `받은 음성함`
- `실행 보드`
- KPI 카드 4개
- 긴 설명 문서 최소화

---

## 7-2. Google Workspace

### 역할
- Google 기반 실행 팀을 위한 대체 운영 공간

### 최종 방향
- Google Sheets를 메인 실행 관제탑으로 둔다
- Docs / Calendar / Gmail은 row 기반 실행 자산으로 연결

### 구조
- 받은 음성함
- 실행 보드
- 설정

### UI 방향
- 시트 셀을 예쁘게 꾸미는 방식은 실패로 판단
- Apps Script HTML Modal Dashboard 사용

---

## 7-3. KakaoTalk

### 상태
- 아직 메인 연결 대상 아님

### 이유
- 비즈니스 계정 / 알림톡 템플릿 / 공급사 절차가 먼저 필요

### 원칙
- Kakao는 나중에 붙인다
- Notion / Google Workspace를 먼저 안정화한다

---

## 8. 받은 음성함과 실행 보드의 차이

## 받은 음성함
- 들어온 음성 입력을 검토하는 곳
- 요약, 화자, 우선순위, 상태, 녹음일 중심
- 아직 실행으로 확정되지 않은 상태

## 실행 보드
- 실제 해야 할 일만 관리하는 곳
- 할 일, 담당자, 상태, 우선순위, 마감일 중심
- 실제 운영과 완료 관리의 중심

핵심 원칙:
- 받은 음성함은 검토용
- 실행 보드는 실행용

---

## 9. 제품이 잘못 가는 신호

아래 중 하나라도 보이면 VOXERA 방향이 흐려진 것이다.

- 단순 기록 저장만 늘어나고 실행 전환이 약해짐
- 설명 문서나 위키가 메인 화면을 잡아먹음
- 사용자가 어디서 뭘 해야 하는지 헷갈림
- 도구 연결이 많아졌는데 실제 실행 속도는 느려짐
- 중복 입력/중복 업로드/중복 보드 생성이 생김
- 초보 사용자가 첫 사용 때 바로 못 씀

---

## 10. Manus가 VOXERA 작업 시 반드시 지켜야 할 기준

1. 항상 실행 중심으로 판단할 것
2. 추상적인 조언보다 바로 적용 가능한 결과물을 우선할 것
3. 사용자 마찰이 적은 구조를 먼저 추천할 것
4. 복잡한 구조보다 단순한 구조를 우선할 것
5. 한국 사용자 기준 직관성을 매우 중요하게 볼 것
6. UI/온보딩/대시보드는 밝고 정돈된 톤으로 설계할 것
7. 비교 시 반드시 추천안 / 비추천안 / 이유를 분명하게 나눌 것
8. 자동화/연결 구조를 제안할 때는 안정성, 유지보수성, 멱등성을 우선할 것
9. 결과물은 항상 복붙/전달/실행 가능한 수준으로 만들 것
10. “사용자가 실제로 행동하게 만드는가”를 최종 판단 기준으로 삼을 것

---

## 11. VOXERA의 한 줄 운영 원칙

**음성을 남기는 것에서 끝나지 말고, 실제 실행이 끝날 때까지 연결한다.**
